## Reuse Strategy

1. Keep enrichment inside scheduled update jobs only. The browser must read static JSON snapshots and must not call enrichment or external maritime APIs.
2. Use existing reference CSV enrichment first, then existing Supabase `vessel_master` cache enrichment, then existing scoring and candidate generation.
3. Preserve enriched fields when writing `vessel_master`: current payloads must not erase known IMO, call sign, operator, owner, manager, type, GT, DWT, or flag values with blanks.
4. Keep `daily:enrich` as a supplemental recovery path for critical missing identity data and historical match memory.
5. Surface enriched fields only through the existing `vessel_display` contract.

## Repair Applied

- `scripts/lib/db.js` now keeps DWT, flag, owner, manager, technical manager, operator source, and enrichment source fields in compact storage payloads.
- Existing `vessel_master` payload is now read during merge and preserved when the new run has blank fields.

- `scripts/audit-enrichment.js` now reports before/after coverage and source contribution by `source.csv`, `vessel_spec`, `AIS info`, `vessel_master_cache`, and port/berth enrichment.

## Remaining Notes

- The current source rows do not always retain a full pre-enrichment copy. When no raw/pre-enrichment payload is available, `audit:enrichment` explicitly reports that the before snapshot basis is limited.
- `vessel_spec` and `mof_ais_info` are treated as existing source labels/configured source paths. This audit does not invent or call any new external API.
- Missing enriched fields should remain visible as `-`; rows must not be hidden only because company or IMO data is incomplete.
